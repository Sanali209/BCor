**РАЗВЕРНУТАЯ СПЕЦИФИКАЦИЯ АРХИТЕКТУРНОГО ЯДРА СИСТЕМЫ**

Данный документ описывает спецификацию архитектурного ядра приложения, построенного на принципах предметно-ориентированного проектирования (DDD), разделения ответственности команд и запросов (CQRS) и событийно-ориентированной архитектуры (EDA). 

Ядро спроектировано для обеспечения слабой связанности (Low Coupling) и высокой сплоченности (High Cohesion) компонентов, скрывая инфраструктурную сложность от бизнес-логики [1, 2].

---

### 1. Фундаментальные архитектурные принципы

*   **Инверсия зависимостей (Dependency Inversion Principle, DIP):** Модули верхнего уровня (предметная область, бизнес-логика) не зависят от модулей нижнего уровня (базы данных, фреймворки, внешние API) [3]. Детали реализации (ORM, веб-контроллеры) зависят от абстракций (интерфейсов), определенных внутри ядра [4]. В Python абстракции реализуются через абстрактные базовые классы (`abc.ABC`) или протоколы (`typing.Protocol`) [5, 6].
*   **Слабая связанность через события (Event-Driven Architecture):** Сервисы и модули взаимодействуют друг с другом преимущественно асинхронно через обмен сообщениями (событиями), что предотвращает создание «распределенного комка грязи» и повышает отказоустойчивость [4, 7, 8].
*   **CQRS (Command Query Responsibility Segregation):** Жесткое разделение путей изменения состояния (Команды/Запись) и получения данных (Запросы/Чтение) [9, 10].

---

### 2. Спецификация слоев и компонентов

Ядро диктует строгое разделение приложения на слои для изоляции обязанностей [11, 12].

#### 2.1. Предметная область (Domain Model)
Слой, содержащий бизнес-правила. Не имеет никаких зависимостей от инфраструктуры [13].
*   **Сущности и Объекты-значения:** Базовые строительные блоки бизнес-логики [14, 15].
*   **Агрегаты (Aggregates):** Кластеры логически связанных объектов, выступающие единой границей согласованности [10, 16, 17]. 
    *   *Спецификация:* Транзакция должна изменять состояние строго *одного* агрегата [18, 19]. Агрегат инкапсулирует логику валидации инвариантов и отклоняет невалидные изменения [20].
*   **События предметной области (Domain Events):** Простые структуры данных (например, `dataclasses`), описывающие факты, произошедшие внутри агрегата [10, 21]. Агрегаты не публикуют события сами, а лишь накапливают их в своем внутреннем списке (например, `self.events`) [22].

#### 2.2. Инфраструктурные абстракции доступа к данным
Ядро предоставляет интерфейсы для работы с данными, скрывая ORM (например, SQLAlchemy) и SQL.

*   **Репозиторий (Repository Pattern):** Абстракция над системой постоянного хранения, предоставляющая иллюзию коллекции объектов в памяти [23-25].
    *   *Спецификация:* Репозитории должны извлекать и сохранять **только Агрегаты** [26, 27]. Интерфейс содержит минимальный набор методов (обычно `add()` и `get()`) [28]. Репозиторий *никогда* не вызывает фиксацию транзакции (`commit`).
*   **Единица работы (Unit of Work, UoW):** Абстракция над атомарными операциями и транзакциями базы данных [29, 30].
    *   *Спецификация:* Реализуется через контекстные менеджеры Python (`__enter__` и `__exit__`) [31, 32]. 
    *   UoW предоставляет доступ к репозиториям (например, `uow.products`) [33, 34].
    *   UoW гарантирует безопасность по умолчанию: при выходе из блока `with` с ошибкой автоматически вызывается `rollback()` [35, 36]. Разработчик должен явно вызывать `uow.commit()` для фиксации [33, 35].
    *   **Сбор событий:** При вызове фиксации, UoW запрашивает у репозитория все загруженные агрегаты, извлекает из них новые события предметной области и передает их в шину сообщений [37, 38].

#### 2.3. Сервисный слой (Service Layer / Handlers)
Определяет варианты использования (Use Cases) системы и оркестрирует инфраструктуру [39, 40].
*   **Обработчики (Handlers):** Принимают Команды или События и экземпляр UoW [41].
*   *Спецификация жизненного цикла:* Обработчик открывает транзакцию (`with uow:`), использует репозиторий для получения агрегата, вызывает доменный метод агрегата и фиксирует транзакцию (`uow.commit()`) [42, 43].

#### 2.4. Шина сообщений (Message Bus)
Центральный диспетчер внутри ядра (например, библиотека `bubus`), маршрутизирующий сообщения к обработчикам [44, 45].

Ядро строго разделяет два типа сообщений (DTO):
*   **Команды (Commands):** Выражают намерение изменить состояние (повелительное наклонение, например `CreateUser`). 
    *   *Спецификация:* Маршрутизируются строго к **одному** обработчику [46, 47]. Ошибки обработки команд прерывают выполнение и возвращаются вызывающей стороне [46, 48].
*   **События (Events):** Факт прошлого (прошедшее время, например `UserCreated`).
    *   *Спецификация:* Транслируются **нулю или множеству** заинтересованных обработчиков (Publish-Subscribe) [48, 49]. Ошибки в обработчиках событий логируются, но не прерывают основной поток выполнения (изоляция отказов) [50, 51].
*   **Мидлвары (Middlewares):** Шина поддерживает внедрение промежуточного ПО для сквозного функционала (логирование, сбор метрик OpenTelemetry, сохранение истории (Event Sourcing / WAL), генерация авто-событий при ошибках) [44].

#### 2.5. Модель Чтения (CQRS Read Model)
Для обеспечения высокой производительности чтения ядро обходит сложную доменную модель и ORM.
*   *Спецификация:* Запросы (Queries) обращаются к оптимизированным представлениям (Projections) или денормализованным таблицам [10, 52].
*   Модели чтения обновляются асинхронно через обработчики событий (например, событие `OrderAllocated` асинхронно обновляет кэш в Redis или таблицу статистики) [10, 53].

### 3. Композитный корень (Bootstrap & Dependency Injection)
Ядро запрещает неявное создание инфраструктурных зависимостей (например, жесткий импорт коннекта к БД внутри бизнес-логики) [54].
*   **Сценарий начальной загрузки (Bootstrap):** Единая точка (Composition Root), где абстракции связываются с их конкретными реализациями [55, 56].
*   *Спецификация:* Загрузчик инициализирует реальные объекты (например, `SqlAlchemyUnitOfWork`) или их подделки (для тестов), внедряет их в функции-обработчики (через замыкания, классы или инспекцию сигнатур `inspect.signature`) и возвращает готовую к работе шину сообщений [57-59].
*   Внешние контроллеры (API Gateway, FastAPI, потребители Kafka/NATS) вызывают только настроенную шину сообщений, передавая ей Команды [60].

### 4. Паттерны отказоустойчивости и распределенной работы
Для поддержки высоконагруженных распределенных сред ядро регламентирует следующие механизмы:
*   **Идемпотентность:** Все обработчики сообщений должны быть спроектированы идемпотентно, чтобы повторная доставка одного и того же события (At-Least-Once delivery) не ломала состояние системы [61-63].
*   **Outbox / Saga:** Для надежной интеграции с внешними брокерами (Kafka, NATS, RabbitMQ) используется паттерн Saga (управление компенсирующими транзакциями при сбоях) [64, 65] или Event Sourcing, где события сначала атомарно пишутся в БД, а затем ретранслируются во внешний мир [16, 66].
*   **Retry / Circuit Breaker:** Для обработки нестабильности внешних API применяются политики повторных попыток (например, с экспоненциальной задержкой через библиотеку `tenacity`) и паттерн Circuit Breaker («Предохранитель»), временно блокирующий запросы к упавшему узлу [67-69].

---

### 5. Стратегия тестирования (Testing Pyramid)
Ядро обеспечивает максимальную тестируемость системы «от края до края» (edge-to-edge) без необходимости использования медленных баз данных или моков (фреймворков имитации) [70].
*   **Подделки (Fakes) вместо моков:** Для тестирования сервисного слоя используются легковесные реализации `FakeRepository` (сохраняющие данные в `set` или `dict` Python) и `FakeUnitOfWork` [25, 71, 72]. Это позволяет тестировать бизнес-логику на огромной скорости, сохраняя реальные паттерны взаимодействия.
*   **Интеграционные тесты:** Используются точечно, только для проверки адаптеров (ORM, брокеры сообщений) на взаимодействие с реальной инфраструктурой (тестовой БД/Docker) [73, 74].
