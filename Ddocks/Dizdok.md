**ДИЗАЙН-ДОКУМЕНТ: Ядро и структура монорепозитория для модульного монолита**

**1. Архитектурная парадигма и стиль**
Система проектируется как **Модульный монолит (Modular Monolith)**, управляемый событиями (Event-Driven Architecture). Данный подход позволяет сохранить простоту единого развертывания и избежать сетевых издержек на начальном этапе, но при этом обеспечивает строгую логическую изоляцию бизнес-доменов (модулей) [1], [2], [3]. 

Фундаментом системы выступает **Предметно-ориентированное проектирование (DDD)** и **Разделение ответственности команд и запросов (CQRS)** [4], [5]. Ядро системы отвечает за инверсию зависимостей (Dependency Inversion): высокоуровневые модули (бизнес-логика) не зависят от низкоуровневых деталей (СУБД, веб-фреймворков), а опираются на абстракции, определенные в ядре [6], [7].

**2. Структура монорепозитория**
Проект использует компоновку `src-layout`, которая надежно разделяет исходный код приложения и тесты, предотвращая ошибки импорта и облегчая упаковку [8], [9].

```text
.
├── docker-compose.yml       # Конфигурация контейнеров (БД, Redis, NATS, воркеры) [10], [11]
├── Makefile                 # Команды для сборки, тестирования и линтинга [10], [12]
├── src/                     # Исходный код пакета приложения [8]
│   ├── setup.py             # Скрипт для локальной установки пакета (pip install -e .) [8], [13]
│   ├── config.py            # Централизованная конфигурация через переменные окружения [14]-[15]
│   ├── core/                # ЯДРО СИСТЕМЫ (Инфраструктурные абстракции)
│   │   ├── messagebus.py    # Реализация шины на базе bubus [16], [17]
│   │   ├── unit_of_work.py  # Абстрактный и реальный классы паттерна UoW [18], [19]
│   │   ├── repository.py    # Абстракции репозиториев (порты) [20]
│   │   └── bootstrap.py     # Композитный корень (Dependency Injection) [21], [22]
│   ├── modules/             # БИЗНЕС-МОДУЛИ (Ограниченные контексты)
│   │   └── orders/          
│   │       ├── domain.py    # Агрегаты, Сущности, Объекты-значения [23]
│   │       ├── commands.py  # Команды (изменение состояния) [24]
│   │       ├── events.py    # События (свершившиеся факты) [25]
│   │       └── handlers.py  # Сервисный слой (обработчики команд/событий) [26]
│   ├── adapters/            # ВТОРИЧНЫЕ АДАПТЕРЫ (Инфраструктура)
│   │   ├── orm.py           # Декларативное отображение SQLAlchemy [27]
│   │   ├── redis_pubsub.py  # Интеграция с внешним брокером [28]
│   │   └── taskiq_broker.py # Интеграция TaskIQ + NATS [29], [30]
│   └── entrypoints/         # ПЕРВИЧНЫЕ АДАПТЕРЫ (Точки входа)
│       ├── fastapi_app.py   # REST/FastAPI контроллеры [31]
│       └── worker.py        # Воркеры TaskIQ для обработки тяжелых задач [32]
└── tests/                   # ТЕСТОВАЯ ПИРАМИДА [8]
    ├── conftest.py          # Общие фикстуры [8]
    ├── unit/                # Быстрые тесты с FakeRepository и FakeUoW [33]
    ├── integration/         # Тесты связи с реальной БД и NATS [34], [35]
    └── e2e/                 # Сквозные тесты через API [36]
```

**3. Архитектура и компоненты Ядра (The Core)**
Ядро инкапсулирует сложность работы с базой данных и маршрутизацией, предоставляя бизнес-модулям чистый декларативный API.

*   **Шина сообщений (Message Bus):** Реализована на базе высокопроизводительной библиотеки `bubus` с поддержкой Pydantic [16], [37]. Шина выступает главным диспетчером и жестко разделяет два типа сообщений:
    *   **Команды (Commands):** Выражают намерение (например, `CreateOrder`). Маршрутизируются строго к *одному* обработчику [38], [39]. Исключения при обработке команд прерывают выполнение и возвращаются пользователю [39], [40].
    *   **События (Events):** Описывают свершившийся факт (например, `OrderCreated`). Могут иметь *множество* обработчиков [41], [42]. Ошибки в обработчиках событий логируются, но не прерывают основной бизнес-процесс [42], [43].
*   **Единица работы (Unit of Work - UoW):** Паттерн, абстрагирующий идею атомарных операций [44]. Реализуется через контекстный менеджер (методы `__enter__` и `__exit__`), обеспечивая безопасный откат (`rollback`) при ошибках или фиксацию (`commit`) при успехе [45]. UoW также отвечает за сбор новых доменных событий из агрегатов (`collect_new_events()`) и передачу их в шину сообщений [46], [47].
*   **Репозиторий (Repository):** Скрывает детали доступа к данным (SQLAlchemy), создавая иллюзию работы с коллекцией в оперативной памяти [48]. Важное правило архитектуры: **один агрегат = один репозиторий**; репозитории извлекают и сохраняют только агрегаты (корневые сущности), а не их внутренние части [49].
*   **Агрегаты (Aggregates):** Являются границами согласованности предметной области [23]. Они инкапсулируют бизнес-правила и защищают инварианты при конкурентном доступе (например, с использованием номеров версий для оптимистичной блокировки) [50], [51]-[52].
*   **Загрузчик / Композитный корень (Bootstrap):** Модуль `bootstrap.py` — это единственное место в системе, где абстрактные интерфейсы связываются с конкретными реализациями инфраструктуры [21], [53]. Он выполняет ручное внедрение зависимостей (Dependency Injection) в обработчики (используя замыкания, классы или `inspect.signature`) и возвращает готовую к работе шину сообщений [54], [22]-[55]. Точки входа (например, FastAPI) просто вызывают `bus = bootstrap.bootstrap()` [56].

**4. Распределенное выполнение и Интеграция**
Для долгих вычислений и связи с внешним миром модульный монолит расширяется с помощью внешних брокеров:
*   **Фоновые задачи:** Интеграция пакетов `TaskIQ` и `NATS` позволяет вынести тяжелые задачи (отправка писем, генерация PDF) из синхронного цикла событий FastAPI во внешние воркеры [29], [57]. TaskIQ оборачивает обработчики и использует механизмы NATS JetStream для балансировки нагрузки, гарантированной доставки сообщений и настройки групп потребителей (Consumer Groups) [29], [32], [30].
*   **Событийно-ориентированная интеграция:** Если микросервису/модулю необходимо сообщить внешним системам об изменениях, внутренние доменные события транслируются в публичные интеграционные события (Integration Events) и публикуются в брокер сообщений (NATS/Redis) с помощью адаптера `EventPublisher` [58], [59]-[60].

**5. Разделение команд и запросов (CQRS)**
Так как сложная предметная область плохо оптимизирована для чтения, в ядре применяется паттерн CQRS:
*   **Модель записи (Write Model):** Обрабатывает *Команды* через строгие агрегаты, репозиторий и UoW для защиты целостности данных [61], [62].
*   **Модель чтения (Read Model):** Обрабатывает *Запросы*, минуя сложные агрегаты и ORM. Используются "сырые" оптимизированные SQL-запросы или отдельные денормализованные таблицы/представления (Views), которые асинхронно обновляются обработчиками событий (например, `Allocated` -> обновить кэш) [63], [64]-[65].

**6. Наблюдаемость и отладка (Observability)**
Для контроля за распределенными и асинхронными процессами ядро использует сквозную телеметрию:
*   **Трассировка:** Интеграция `OtelTracingMiddleware` из `bubus` создает спаны OpenTelemetry для каждого события и связывает родительские/дочерние вызовы в единый трейс [66].
*   **Логирование:** В каждый обработчик интегрировано структурированное логирование (например, через `Loguru` или `LoggerEventBusMiddleware` в `bubus`), которое фиксирует входные параметры (`event` или `command`), что критически важно для восстановления контекста при сбоях в асинхронной среде [66], [67]-[68].

**7. Пирамида тестирования (TDD)**
Архитектура позволяет проводить сквозное тестирование без поднятия базы данных благодаря инверсии зависимостей [69]:
*   **Уровень Unit-тестов:** Тестирование сервисного слоя (хэндлеров) и доменной модели в изоляции с использованием `FakeRepository` и `FakeUnitOfWork` (сохраняющих состояние в простых списках или словарях Python) [70], [71], [72].
*   **Интеграционные тесты:** Точечная проверка того, как конкретные адаптеры (например, `SqlAlchemyRepository` или `NATS Broker`) общаются с реальной инфраструктурой [73], [74].
*   **Сквозные тесты (E2E):** Тестирование "от края до края" через API с реальной БД и шиной сообщений [36].
