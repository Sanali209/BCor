# Roadmap: TDD-Разработка недостающих компонентов архитектурного ядра (BCor)

## Методология формирования и исполнения Roadmap

Данный Roadmap строится и исполняется в соответствии с инженерными практиками, обеспечивающими высокое качество и предсказуемость результата. 

### 1. Формирование Roadmap
*   **Изучение текущей кодовой базы (As-Is):** Перед планированием проводится аудит существующего кода и `tests/`, чтобы исключить дублирование работы (например, если `FakeRepository` уже написан).
*   **Декомпозиция на Атомарные Задачи:** Каждая Фаза разбивается на шаги (задачи), которые физически выполнимы за один подход разработки. Задача содержит входные данные (какие абстракции использовать) и критерии приемки.

### 2. Исполнение Roadmap
*   **Режим планирования для каждой фазы (implementation_plan):** Перед стартом любой новой фазы пишется детальный пошаговый план (`implementation_plan.md`), описывающий точные изменения в файлах и последовательность TDD и согласуется с разработчиком.
*   **Test-Driven Development (TDD):**
    1.  **Red:** Сначала пишется падающий тест (unit-тест с `FakeUoW` для бизнес-логики или integration-тест со SQLite для инфраструктуры).
    2.  **Green:** Пишется минимально достаточный код для успешного прохождения теста.
    3.  **Refactor:** Улучшение структуры кода, применение линтеров, исправление типизации.
*   **Интеграционные проверки (Phase Checks):** Каждая фаза **обязана** закрываться задачей проверки:
    *   *Линтеры и Форматтеры:* `ruff check`, `ruff format`, `flake8`, `mypy`. В ядре не должно накапливаться "технического долга" по стилю кода.
    *   *Тестирование на деградацию:* Запуск всего пакета `pytest`, чтобы убедиться, что новый код не сломал функционал предыдущих фаз. Только при $100\%$ покрытии и успешном прохождении фаза считается закрытой.


Настоящий план (Roadmap) описывает последовательность создания механизмов, описанных в документации, но **еще не реализованных** в кодовой базе (с учетом того, что базовое ядро, модули `ecs`, `agm`, `analytics` и тестовая инфраструктура с `FakeRepository`/`FakeUnitOfWork` уже существуют).
Разработка каждой фичи должна вестись строго по **Test-Driven Development (TDD)**:
1. Пишется падающий тест (Red).
2. Пишется минимальный код для его выполнения (Green).
3. Производится рефакторинг (Refactor).

## Фаза 5: Интеграция AGM с Архитектурным Ядром (Refactoring)
**Цель:** Оформить Aetheris Graph Mapper (AGM) как полноценный модуль (без привязки к Repository/UoW, т.к. это инфрастуктурный маппер), устранив прямые вызовы TaskIQ в пользу Event-Driven подхода.

*   - [x] **Задача 5.1: Устранение Leaky Abstraction (Event Routing)**
    *   *Данные для исполнения:* Создать `messages.py` в модуле `agm`. Определить событие (например, `StoredFieldRecalculationRequested`). Изменить `AGMMapper.save()`: вместо прямого вызова `.kiq()`, маппер должен возвращать или публиковать список сгенерированных событий изменения. Написать обработчик, который слушает это событие и дергает TaskIQ.
    *   *TDD Тест:* Интеграционный тест, проверяющий, что при сохранении объекта со `Stored` полем, генерируется правильное событие.
*   - [x] **Задача 5.2: Регистрация `AGMModule`**
    *   *Данные для исполнения:* Написать `src/modules/agm/module.py` (BaseModule). Зарегистрировать провайдер `AGMMapper` и подписать обработчик TaskIQ на новое событие.
*   - [x] **Задача 5.3: Проверки Фазы 5**
    *   *Линтеры и Форматтеры:* `ruff`, `flake8`, `mypy`.
    *   *Деградация:* Прогон `pytest tests`. Должны работать тесты `agm`, а также существующие модули.
