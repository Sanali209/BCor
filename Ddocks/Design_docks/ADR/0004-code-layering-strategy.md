# ADR 0004: Стратегия разделения уровней кода и монорепозитория

## Статус
Принято (Accepted)

## Контекст
С ростом экосистемы BCor возникает необходимость четкого разделения ответственности между различными частями кодовой базы. Без единого стандарта существует риск "размывания" логики приложения по ядру системы или, наоборот, дублирования инфраструктурного кода в разных приложениях.

## Решение
Мы внедряем 4-уровневую структуру организации кода в монорепозитории:

### 1. Ядро (Core — `src/core`)
*   **Назначение**: Обеспечение базовых механизмов функционирования системы.
*   **Что включает**: `MessageBus`, система обнаружения модулей (`ModuleDiscovery`), интеграция DI (`Dishka`), базовые типы сообщений и монады (`BusinessResult`).
*   **Правило**: Не содержит бизнес-логики. Зависит только от фундаментальных библиотек (`pydantic`, `dishka`, `asyncio`).

### 2. Системные модули (Modules — `src/modules`)
*   **Назначение**: Предоставление общих инфраструктурных сервисов.
*   **Что включает**: Коннекторы к БД (Generic SQL), системы логирования, общие UI-компоненты (Shared Widgets), визуализаторы (Plotly Helpers).
*   **Правило**: Модули независимы друг от друга. Подключаются приложениями через конфигурацию.

### 3. Приложения (Apps — `src/apps`)
*   **Назначение**: Реализация конкретных бизнес-задач пользователя.
*   **Что включает**: `ImageAnalyze`, `VFSSample` и другие. Каждое приложение имее свою внутреннюю структуру (Domain, Use Cases, Adapters).
*   **Правило**: Приложения могут зависеть от `Core` и `Modules`, но не друг от друга.

### 4. Общие библиотеки (Common/Lib — `src/common`)
*   **Назначение**: Хранение чистого кода (Pure Python), не привязанного к фреймворку BCor.
*   **Что включает**: Математические утилиты, алгоритмы обработки данных, регулярные выражения.

## Принципы развития
### Принцип Experimental First
Все новые фичи или паттерны, извлекаемые из конкретного приложения, сначала реализуются на **экспериментальном уровне** непосредственно внутри кодовой базы этого приложения (в папке `core` или `internal`).
*   **Локальность**: Код остается в приложении до подтверждения его полезности.
*   **Валидация**: Фича должна пройти проверку в реальных условиях эксплуатации приложения.
*   **Миграция**: Только после явного одобрения и подтверждения архитектурной ценности код переносится в `src/core` или `src/modules` для использования всей платформой.

## Последствия
*   **Положительные**:
    *   Высокая степень переиспользования кода через общие модули.
    *   Упрощенное тестирование: ядро и модули тестируются независимо.
    *   Легкое внедрение новых разработчиков: понятно, куда класть код.
*   **Отрицательные**:
    *   Требуется строгая дисциплина при рефакторинге (правило "трёх").
    *   Немного более сложная структура импортов.

## Паттерны для извлечения
1.  **Thread Bridge**: Перенос асинхронных воркеров для GUI в `src/core/gui`.
2.  **Generic Repository**: Создание шаблонов CRUD операций в `src/modules/storage`.
3.  **Theme Provider**: Вынос общих стилей PySide6 в `src/modules/ui`.
