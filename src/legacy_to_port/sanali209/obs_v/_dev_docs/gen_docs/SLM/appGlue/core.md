# Core Application Glue (`core.py`)

Этот файл является сердцем модуля `appGlue` и определяет основные классы и механизмы, связывающие различные части приложения SLM. Он включает системы для управления конфигурацией, ресурсами (сервисами), модулями, событиями и бэкендами.

## Зависимости

*   `dependency_injector`: Используется для импорта, но не применяется в коде напрямую.
*   `SLM.appGlue.modularity.modulemanager.ModuleManager`: Импортируется, но не используется напрямую в этом файле.
*   `SLM.appGlue.DesignPaterns.MessageSystem.MessageSystem`: Используется системой конфигурации для уведомлений об изменениях.
*   `typing`: Для аннотаций типов (`TypeVar`, `Type`, `List`).

## Основные компоненты

### 1. Система Конфигурации

Эта система позволяет управлять конфигурацией приложения, разделенной на секции.

*   **`ConfigStorage`**: Базовый абстрактный класс для хранилища конфигурации. Конкретные реализации (например, `TestConfigStorage` или хранилище на основе файла) должны наследоваться от него.
    *   `load()`: Загружает конфигурацию.
    *   `save(section_name, attr_name, value)`: Сохраняет значение.
*   **`BaseConfig`**: Основной класс для доступа к конфигурации. Использует `ConfigStorage`.
    *   Доступ к секциям через атрибуты (например, `config.database`).
    *   Автоматически создает экземпляры `ConfigSection` при первом доступе.
    *   `register_section(section_name, section_class)`: Позволяет зарегистрировать типизированную секцию (`TypedConfigSection`).
    *   `load_config()`: Загружает конфигурацию из хранилища.
    *   `notify_change(section_name, attr_name, value)`: Уведомляет об изменении через `MessageSystem` и сохраняет значение в хранилище.
*   **`ConfigSection`**: Представляет секцию конфигурации. Доступ к параметрам секции через атрибуты.
*   **`TypedConfigSection`**: Расширение `ConfigSection` для секций с предопределенными атрибутами и типами (хотя проверка типов не реализована явно).
*   **`ConfigListener`**: Базовый класс для компонентов, которые должны реагировать на изменения конфигурации. Подписывается на сообщение `"ConfigChanged"` от `MessageSystem`. #todo change to event use message sistem not so good
    *   `on_config_change(self, section, attribute, value)`: Метод, вызываемый при изменении конфигурации.

### 2. Управление Ресурсами/Сервисами

Система для регистрации и получения доступа к общим ресурсам или сервисам приложения.

*   **`Resource`**: Базовый класс для любого ресурса, управляемого `ResourcesRepo`. Имеет атрибуты `name` и `tags`.
*   **`ResourcesRepo`**: Репозиторий (реестр) для хранения и управления экземплярами `Resource`.
    *   `register(resource: Resource)`: Регистрирует экземпляр ресурса.
    *   `register_or_override(_type, instance: Resource)`: Регистрирует или заменяет существующий ресурс того же типа.
    *   `get_by_type_one(_type: Type[T]) -> T`: Возвращает первый зарегистрированный ресурс указанного типа.
    *   `get_by_name_one(name: str) -> Resource`: Возвращает ресурс по имени.
    *   `has_service(_type: Type[T]) -> bool`: Проверяет наличие ресурса указанного типа.
*   **`Service`**: Базовый класс для сервисов, наследуется от `Resource`. Сервисы имеют жизненный цикл (`init`, `dispose`, `lasy_load`).
    *   `init(config)`: Инициализация сервиса (получает конфигурацию).
    *   `dispose()`: Освобождение ресурсов сервиса.
    *   `lasy_load()`: Метод для отложенной инициализации (вызывается `Allocator` при первом запросе).
    *   `instance() -> T` (статический): Удобный метод для получения экземпляра сервиса через `Allocator`.

### 3. Система Модульности

Позволяет структурировать приложение в виде модулей с зависимостями и жизненным циклом.

*   **`Module`**: Базовый класс для модуля приложения.
    *   `name` (str): Имя модуля.
    *   `dependencies` (List[str]): Список имен модулей, от которых зависит данный модуль (не используется в текущей логике `Allocator`).
    *   `enabled` (bool): Флаг, включен ли модуль.
    *   `init()`: Метод для инициализации модуля (обычно здесь регистрируются сервисы в `Allocator`).
    *   `load()`: Метод для загрузки ресурсов модуля после инициализации всех сервисов.
    *   `unload()`: Метод для освобождения ресурсов модуля при завершении работы.

### 4. Центральный `Allocator` (Распределитель/Локатор Сервисов)

Статический класс, который управляет конфигурацией, модулями и ресурсами/сервисами.

*   **Атрибуты класса:**
    *   `config` (`BaseConfig`): Экземпляр системы конфигурации.
    *   `modules` (List[`Module`]): Список зарегистрированных модулей.
    *   `res` (`ResourcesRepo`): Экземпляр репозитория ресурсов/сервисов.
*   **Методы класса:**
    *   `add_module(module: Module)`: Регистрирует модуль.
    *   `disable_module(name: str, value: bool = False)`: Включает/выключает модуль по имени.
    *   `get_instance(key: Type[T]) -> T`: Получает экземпляр сервиса/ресурса по типу из `ResourcesRepo`. Выполняет отложенную загрузку (`lasy_load`) для сервисов при первом доступе.
    *   `init_modules()`: Инициализирует все включенные модули (`module.init()`), затем инициализирует все зарегистрированные сервисы (`service.init()`), а затем загружает модули (`module.load()`).
    *   `init_services()`: Загружает конфигурацию и вызывает `init()` у всех зарегистрированных сервисов.
    *   `dispose_services()`: Вызывает `dispose()` у всех сервисов и `unload()` у всех включенных модулей.

### 5. Система Событий (`Event`)

Простая реализация паттерна "Наблюдатель" (Publish/Subscribe).

*   `subscribe(callback)`: Подписка на событие.
*   `unsubscribe(callback)`: Отписка от события.
*   `fire(*args, **kwargs)`: Инициирование события, вызов всех подписчиков.
*   Поддерживает операторы `+=` и `-=` для подписки/отписки.

### 6. Управление Бэкендами

Система для управления различными реализациями (бэкендами) для определенного типа функциональности (например, разные модели для генерации эмбеддингов).

*   **`ServiceBackend`**: Базовый класс для бэкенда.
    *   `name` (str): Уникальное имя бэкенда.
    *   `lazy_loaded` (bool): Флаг отложенной загрузки.
    *   `tags` (list): Теги для категоризации бэкендов.
    *   `is_compatible(*args, **kwargs)`: Проверка совместимости (не реализована).
    *   `load(*args, **kwargs)`: Метод для загрузки ресурсов бэкенда (вызывается при первом доступе).
*   **`BackendProvider`**: Репозиторий (реестр) для бэкендов определенного типа, наследуется от `Resource`.
    *   `register_backend(backend: ServiceBackend)`: Регистрирует бэкенд.
    *   `get_backend_by_name(name: str) -> ServiceBackend`: Получает бэкенд по имени, выполняя отложенную загрузку (`load()`) при необходимости.
    *   `get_all_backends()`: Возвращает список имен всех зарегистрированных бэкендов.
    *   `get_all_by_tag(tag: str)`: Возвращает имена бэкендов с указанным тегом.

### 7. Класс Приложения (`GlueApp`)

Основной класс приложения.

*   Наследуется от `Resource`.
*   Хранит экземпляр `BaseConfig`.
*   Регистрирует себя в `Allocator`.
*   `init()`: Инициирует запуск инициализации модулей через `Allocator`.
*   `exit()`: Инициирует освобождение ресурсов через `Allocator`.
*   `run()`: Метод для основного цикла приложения (не реализован).

### 8. Контекст Приложения (`ModeManager`, `ContextMode`)

*   **`ModeManager`**: Управляет различными режимами работы приложения.
    *   `modes` (dict): Словарь зарегистрированных режимов.
    *   `current_mode`: Текущий активный режим.
    *   `set_mode(mode_name)`: Переключает приложение в указанный режим, вызывая `deactivate()` у старого режима и `activate()` у нового.
*   **`ContextMode`**: Базовый класс для режима приложения.
    *   `activate()`: Вызывается при активации режима.
    *   `deactivate()`: Вызывается при деактивации режима.

## Взаимодействие компонентов

1.  При старте приложения создается экземпляр `GlueApp`.
2.  `GlueApp` создает `BaseConfig` и регистрирует себя в `Allocator.res`.
3.  Различные части приложения (или конфигурационные файлы) регистрируют `Module` в `Allocator`.
4.  `GlueApp.init()` вызывает `Allocator.init_modules()`.
5.  `Allocator` проходит по включенным модулям и вызывает их метод `init()`.
6.  Модули в своих `init()` регистрируют необходимые `Service` и `BackendProvider` в `Allocator.res`.
7.  `Allocator` загружает конфигурацию (`config.load_config()`) и вызывает `init()` у всех зарегистрированных `Service`.
8.  `Allocator` вызывает `load()` у всех включенных модулей.
9.  Приложение работает. Компоненты получают доступ к сервисам через `Allocator.get_instance(ServiceType)` или `ServiceType.instance()`.
10. `BackendProvider` используется для получения конкретных реализаций (`ServiceBackend`) по имени.
11. Система конфигурации позволяет читать и изменять настройки, уведомляя `ConfigListener` через `MessageSystem`.
12. При завершении `GlueApp.exit()` вызывает `Allocator.dispose_services()`, который вызывает `dispose()` у сервисов и `unload()` у модулей.

Этот фреймворк обеспечивает слабую связанность компонентов через `Allocator` и `ResourcesRepo`, поддерживает модульность, централизованную конфигурацию и управление жизненным циклом сервисов.
