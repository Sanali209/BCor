# Hyperparameter Tuning for Single-Label Classification (`singl_label_tune2.py`)

Этот скрипт расширяет функциональность `singl_label_train.py`, добавляя автоматический **поиск и оптимизацию гиперпараметров** с использованием библиотеки **Ray Tune**. Он позволяет эффективно находить лучшие настройки (скорость обучения, размер батча, тип оптимизатора и т.д.) для модели классификации изображений, обучаемой на аннотациях из SLM.

## Зависимости

*   **Основные ML/DL:** `torch`, `transformers`, `sklearn` (для `train_test_split`), `PIL (Pillow)`.
*   **Hyperparameter Tuning:** `ray`, `ray.tune`, `ray.air`.
*   **Кэширование:** `diskcache`.
*   **Утилиты:** `os`, `random`, `hashlib`, `tqdm`, `loguru`, `tempfile`, `shutil`, `functools.partial`.
*   **Hugging Face Hub:** `huggingface_hub`.
*   **SLM:** `SLM.appGlue.core.Allocator`, `SLM.files_data_cache.thumbnail.ImageThumbCache`, `SLM.files_db.annotation_tool.annotation` (`AnnotationRecord`, `AnnotationJob`).

## Ключевые улучшения и компоненты

1.  **`ImageClassificationDataset(Dataset)` (с кэшированием)**:
    *   Использует `diskcache` для кэширования результатов предобработки изображений (`feature_extractor`).
    *   Ключом кэша служит MD5 хэш пути к файлу.
    *   При запросе элемента (`__getitem__`) сначала проверяется наличие данных в кэше. Если данные есть, они возвращаются.
    *   Если данных нет (cache miss), изображение загружается, обрабатывается `feature_extractor`-ом, результат помещается в кэш (тензоры сохраняются на CPU) и затем возвращается.
    *   Это значительно ускоряет загрузку данных, особенно при многократных прогонах (эпохах) и параллельных испытаниях (trials) в Ray Tune.
    *   Использует `loguru` для логирования попаданий/промахов кэша и ошибок.

2.  **`evaluate_model(...)`**:
    *   Функция оценки оставлена практически без изменений, добавлены лишь информационные логи с помощью `loguru`.

3.  **`train_model_tune(...)` (Адаптировано для Ray Tune)**:
    *   Функция обучения, вызываемая внутри каждого trial Ray Tune.
    *   Цикл обучения по эпохам и батчам остается схожим с `singl_label_train.py`.
    *   **Интеграция с Ray Tune:** После каждой эпохи (и валидации):
        *   Собираются метрики (`val_loss`, `val_acc`, `train_loss`, `train_acc`, `epoch`).
        *   Модель сохраняется во временную директорию (`tempfile.TemporaryDirectory`).
        *   Создается объект `Checkpoint` из этой директории.
        *   Метрики и чекпоинт передаются в Ray Tune с помощью `ray.air.session.report(metrics, checkpoint=checkpoint)`. Ray Tune использует эти данные для принятия решений (например, остановка trial в ASHA) и отслеживания лучшего результата.
    *   Добавлена проверка на NaN/Inf в значении потерь для раннего прерывания "плохих" trials.

4.  **`train_tune_wrapper(config, ...)` (Обертка для Ray Tune Trial)**:
    *   Основная функция, которую Ray Tune запускает для каждого испытания (trial).
    *   Принимает `config` (словарь с гиперпараметрами для данного trial) и ссылки (`ObjectRef`) на данные, предварительно загруженные в Ray Object Store.
    *   **Шаги внутри trial:**
        *   Устанавливает random seed.
        *   Определяет устройство (`cuda` или `cpu`).
        *   Загружает `feature_extractor`.
        *   Получает данные (аннотации, маппинги меток) из Ray Object Store (`ray.get`).
        *   Создает экземпляры `ImageClassificationDataset` (с уникальным путем кэша для каждого trial).
        *   Создает `DataLoader`-ы (пытается оптимизировать `num_workers`).
        *   Загружает базовую модель `AutoModelForImageClassification`.
        *   Настраивает оптимизатор (`AdamW` или `Adam`) и функцию потерь (`CrossEntropyLoss`) на основе `config`.
        *   Вызывает `train_model_tune` для выполнения цикла обучения и репортинга в Ray Tune.
        *   Очищает директорию кэша `diskcache` после завершения trial.

5.  **`predict_image(...)`**:
    *   Функция инференса остается без изменений.

## Основной блок (`if __name__ == "__main__":`) (Оркестрация Ray Tune)

1.  **Инициализация SLM:** Выполняется один раз перед запуском Ray.
2.  **Предварительная загрузка данных:**
    *   Загружаются аннотации из `AnnotationJob` ("NSFWFilter").
    *   Создаются маппинги меток.
    *   Формируется список `records` с путями к миниатюрам и метками.
    *   Данные разделяются на обучающую и валидационную выборки.
    *   **Ключевой шаг:** Подготовленные списки `train_annotations`, `val_annotations` и маппинги помещаются в **Ray Object Store** с помощью `ray.put()`. Это позволяет эффективно передавать большие объемы данных во все worker-ы Ray Tune без копирования.
3.  **Инициализация Ray:** Запускается `ray.init()`, если еще не запущен. Указывается временная директория для Ray.
4.  **Определение пространства поиска:** Задается словарь `search_space`, где ключи - имена гиперпараметров, а значения - распределения или варианты, которые Ray Tune будет пробовать (например, `tune.loguniform`, `tune.choice`).
5.  **Настройка планировщика (Scheduler):** Используется `ASHAScheduler` для ранней остановки неперспективных trials. Он отслеживает `val_acc` и останавливает trials, которые показывают плохие результаты после `grace_period` эпох.
6.  **Создание Trainable:** Функция `train_tune_wrapper` оборачивается с помощью `functools.partial` для передачи ссылок на данные (`*_ref`) в качестве фиксированных аргументов.
7.  **Запуск `tune.Tuner`:**
    *   Конфигурируется `tune.Tuner` для запуска процесса оптимизации.
    *   `tune.with_resources` используется для указания требуемых ресурсов (CPU, GPU) для каждого trial.
    *   `param_space`: Передается пространство поиска.
    *   `tune_config`: Задается количество испытаний (`num_samples`) и планировщик (`scheduler`).
    *   `run_config`: Настраивается имя эксперимента, путь для сохранения результатов (`storage_path`), конфигурация чекпоинтов (`CheckpointConfig` - сохранять лучший по `val_acc`).
    *   Вызывается `tuner.fit()` для старта процесса тюнинга.
8.  **Анализ результатов:**
    *   После `tuner.fit()` извлекаются результаты (`results`).
    *   Находится лучший trial с помощью `results.get_best_result()`.
    *   Выводятся лучшие метрики и соответствующая конфигурация гиперпараметров.
9.  **Сохранение и загрузка лучшей модели:**
    *   Извлекается лучший чекпоинт (`best_result.checkpoint`).
    *   Чекпоинт загружается в локальную директорию (`best_checkpoint.as_directory()`).
    *   Модель `AutoModelForImageClassification` и `feature_extractor` загружаются из директории чекпоинта.
    *   Лучшая модель и экстрактор сохраняются в отдельную папку (`SAVE_DIR_BEST_MODEL`).
    *   **Опционально:** Лучшая модель загружается на Hugging Face Hub (если указаны `HUB_MODEL_ID` и токен `HF_TOKEN` в переменных окружения).
10. **Завершение Ray:** Вызывается `ray.shutdown()`.

## Преимущества использования Ray Tune

*   **Параллельное выполнение:** Ray Tune может запускать множество испытаний с разными гиперпараметрами параллельно на нескольких CPU и GPU.
*   **Эффективное использование ресурсов:** Планировщики, такие как ASHA, позволяют экономить ресурсы, останавливая плохие испытания на ранних стадиях.
*   **Автоматизация:** Весь процесс поиска лучших гиперпараметров автоматизирован.
*   **Масштабируемость:** Ray позволяет легко масштабировать процесс на кластер машин.

Этот скрипт демонстрирует продвинутый подход к обучению моделей, интегрируя сбор данных из SLM, эффективную предобработку с кэшированием и автоматизированную оптимизацию гиперпараметров с помощью Ray Tune.
