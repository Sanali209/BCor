# Документация архитектуры BCor

## Обзор
BCor — это **Модульный Монолит, Управляемый Событиями**, построенный на принципах чистой архитектуры, DDD (предметно-ориентированное проектирование) и CQRS (разделение команд и запросов). Система спроектирована для масштабируемости, поддерживаемости и тестируемости.

## Основные архитектурные принципы

### 1. Слои чистой архитектуры
```
┌─────────────────────────────────────┐
│           Слой представления        │  ← Точки входа (FastAPI, CLI, GUI)
├─────────────────────────────────────┤
│           Слой приложения           │  ← Команды, События, Обработчики
├─────────────────────────────────────┤
│             Слой домена             │  ← Агрегаты, Сущности, Объекты-значения
├─────────────────────────────────────┤
│          Слой инфраструктуры        │  ← Адаптеры, Репозитории, Внешние сервисы
└─────────────────────────────────────┘
```

### 2. Правило зависимостей
- Внутренние слои **никогда** не зависят от внешних слоев
- Слой домена имеет **ноль** внешних зависимостей
- Инфраструктура реализует интерфейсы домена (порты)

### 3. Управляемая событиями коммуникация
- Модули общаются **только** через события через `bubus.EventBus`
- Нет прямых вызовов методов между модулями
- Слабая связанность обеспечивает независимую эволюцию модулей

## Основные компоненты

### Шина сообщений (`src/core/messagebus.py`)
- Обертка над `bubus.EventBus`
- Маршрутизирует Команды к единственным обработчикам
- Транслирует События множеству подписчиков
- Автоматическое DI-внедрение через `dishka`
- Интеграция трассировки OpenTelemetry

### Единица работы (`src/core/unit_of_work.py`)
- Абстракция границ транзакции
- Управляет экземплярами репозиториев
- Собирает доменные события из агрегатов
- Автоматический откат при исключениях

### Паттерн репозитория (`src/core/repository.py`)
- Базовый обобщенный класс `AbstractRepository[T]`
- Отслеживает загруженные агрегаты в множестве `seen`
- Реализуется конкретными адаптерами (SQLAlchemy, JSON и т.д.)

### Композиция системы (`src/core/system.py`)
- Загрузка через `System.from_manifest()`
- Загружает конфигурации `app.toml`
- Инициализирует DI-контейнер с `dishka`
- Регистрирует все обработчики модулей

## Структура модуля

Каждый модуль следует согласованной структуре:
```
src/modules/<module>/
├── __init__.py
├── module.py          # Регистрация модуля
├── messages.py        # Команды и События
├── handlers.py        # Бизнес-логика
├── provider.py        # DI-провайдеры
├── domain/            # Доменная модель
└── adapters/          # Реализации инфраструктуры
```

### Типы модулей
- **Основные модули**: `ecs`, `agm`, `analytics`, `orders`, `vfs`, `llm`, `search`
- **Приложения**: `ImageAnalyze`, `ImageDedup`, `VFSSample`, `hello_app`
- **Общие библиотеки**: `src/common/` (утилиты на чистом Python)

## Пример потока событий

```
1. FastAPI получает POST /orders
2. Создает CreateOrderCommand
3. MessageBus маршрутизирует к handle_create_order
4. Обработчик использует OrderRepository (через DI)
5. Агрегат Order генерирует событие OrderCreated
6. UnitOfWork собирает событие
7. MessageBus публикует OrderCreated
8. Модуль analytics подписывается и логирует
```

## Конфигурация

### Манифест приложения (`app.toml`)
```toml
[modules]
enabled = ["ecs", "analytics", "orders"]

[settings]
# Настройки, специфичные для модуля
```

### Переменные окружения
- Загружаются через `pydantic-settings`
- Быстрый отказ при отсутствии обязательных переменных
- Валидируются при запуске

## Наблюдаемость

### Логирование
- `loguru` заменяет стандартный `logging`
- Структурированный JSON-вывод
- Контекстно-зависимый с trace ID

### Трассировка
- Интеграция OpenTelemetry
- Спаны для каждой команды/события
- Поддержка распределенной трассировки

### Метрики
- Метрики Prometheus в воркерах TaskIQ
- Порт 9000 для эндпоинта метрик

## Стратегия тестирования

### Пирамида тестирования
```
       E2E тесты (Мало)
          ↑
   Интеграционные тесты (Некоторые)
          ↑
    Модульные тесты (Много)
```

### Тестовые заглушки
- `FakeRepository`: Реализация в памяти
- `FakeUnitOfWork`: Нетранзакционные операции
- Находятся в `tests/conftest.py`

### Рабочий процесс TDD
1. Написать падающий тест (Red)
2. Реализовать минимальный код (Green)
3. Рефакторинг для ясности (Refactor)

## Стек технологий

| Компонент | Технология |
|-----------|------------|
| Управление пакетами | `uv` |
| Шина событий | `bubus` |
| DI-контейнер | `dishka` |
| Настройки | `pydantic-settings` |
| Фоновые задачи | `TaskIQ` + `NATS` |
| Обработка ошибок | `returns` (Result Monad) |

| Трассировка | `OpenTelemetry` |
| Метрики | `Prometheus` |
| Тестирование | `pytest`, `pytest-asyncio` |

## Миграция с легаси

### Паттерн "Душитель"
1. Обернуть легаси-приложение в модуль BCor
2. Постепенно извлекать бизнес-логику
3. Заменить вызовы UI командами
4. Удалить легаси-код при изоляции

### Антикоррупционный слой
- ACL-адаптеры мостят легаси-концепции
- Никогда не импортировать легаси-код из новых модулей
- Подход миграции, ориентированный на домен

## Лучшие практики

### Разработка модулей
1. Начните с `messages.py` (определите команды/события)
2. Создайте доменную модель в `domain/`
3. Реализуйте обработчики с бизнес-логикой
4. Зарегистрируйте в `module.py`
5. Пишите тесты первыми (TDD)

### Обработка ошибок
- Используйте монаду `BusinessResult` для бизнес-ошибок
- Исключения для ошибок инфраструктуры
- Быстрый отказ для ошибок конфигурации
- Безопасный отказ для обработчиков событий

### Производительность
- Async/await повсюду
- Фоновые задачи для тяжелых операций
- CQRS read-модели для запросов
- Пул соединений для баз данных
