9.2 Key Technology Decisions Analysis

bubus (Event Bus)

Why chosen: Lightweight asyncio-native event bus with decorator-based routing. Минимальная кодовая база, легко пачится (как сделано в BCor — отключение 2-level loop prevention).

Risk: Small community (обновления редки). BCor уже пачит bubus — если bubus перестанет поддерживаться, придётся форкать.

Alternatives: Встроенный asyncio.Queue для простых случаев, либо полноценная шина типа RabbitMQ/aio-pika для распределённых сценариев.

dishka (DI Container)

Why chosen: Async-native, Scope-управление (APP/REQUEST), минимальный boilerplate. Лучшая альтернатива dependency-injector для asyncio-приложений.

Risk: Низкий (активно развивается, версия 1.9+).

TaskIQ + NATS

Why chosen: Современная asyncio-native замена Celery. NATS — быстрый, лёгкий message broker.

Risk: TaskIQ имеет меньше интеграций, чем Celery. NATS требует отдельного сервера.

Alternative: arq (Redis-based) — проще, Redis часто уже есть в проекте.

returns (Result Monad)

Why chosen: Самая полная реализация функциональных типов для Python: Result, Maybe, IO, Reader, Writer, State.

Risk: Может быть overkill для небольших проектов. Увеличивает сложность для новичков.
