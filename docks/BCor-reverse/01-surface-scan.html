1.4 Configuration Pattern

Config-driven module system. Each application has an app.toml manifest that declares which modules to enable and their settings. The System.from_manifest() method in src/core/system.py reads the manifest, discovers modules via ModuleDiscovery, instantiates them, and builds the DI container with dishka.

Settings are validated via pydantic-settings. Environment variables override TOML values at runtime.

1.6 Extracted Concepts

PATTERN

Event-Driven Modular Monolith

Где: src/core/messagebus.py, src/core/module.py, src/core/system.py

Что: Модули (orders, agm, ecs и др.) изолированы и общаются ТОЛЬКО через события. Никаких прямых вызовов между модулями. Каждый модуль — независимый бизнес-домен.

Зачем: Слабая связанность (loose coupling), возможность независимой эволюции модулей, лёгкое тестирование.

PATTERN

Clean Architecture Layers

Где: src/core/ (domain), src/adapters/ (infrastructure), src/apps/ (entrypoints)

Что: 4 слоя: Domain → Application → Infrastructure → Presentation. Внутренние слои НИКОГДА не зависят от внешних. Domain слой — чистый Python без внешних зависимостей.

Зачем: Тестируемость, независимость от фреймворков, возможность замены инфраструктуры.

PATTERN

CQRS (Command Query Responsibility Segregation)

Где: src/core/messages.py — Command и Event базовые классы

Что: Команды (изменение состояния) направляются строго одному обработчику. События (факты) транслируются всем подписчикам. Команды и события — наследники Message.

PATTERN

Unit of Work + Repository

Где: src/core/unit_of_work.py, src/core/repository.py

Что: UoW координирует транзакции и собирает доменные события из агрегатов. Repository абстрагирует хранение. AbstractRepository[T] generic с трекингом seen агрегатов.

Зачем: Атомарность операций, сбор событий для публикации, тестирование через FakeRepository.

PATTERN

DI Container (dishka)

Где: src/core/system.py — CoreProvider, модульные Provider'ы

Что: Автоматическое внедрение зависимостей в обработчики команд/событий. Контейнер строится при bootstrap'е системы.

Зачем: Никакого ручного получения зависимостей. Handler'ы просто объявляют параметры — dishka внедряет.

PATTERN

Result Monad (returns)

Где: src/common/monads.py — BusinessResult = Result[T, E]

Что: Бизнес-ошибки возвращаются как Failure, успех — как Success. Исключения — только для инфраструктурных ошибок. Fail-fast для конфигов, fail-safe для event handler'ов.

PATTERN

Config-Driven Module Discovery

Где: src/core/discovery.py, файлы app.toml

Что: Модули загружаются динамически из TOML-манифеста. ModuleDiscovery ищет подклассы BaseModule в указанных пакетах.

PATTERN

Lifecycle Hooks (@on_start / @on_stop)

Где: src/core/decorators.py

Что: Декораторы для регистрации функций, вызываемых при старте/остановке системы. Поддержка sync и async.

PATTERN

Strangler Fig (Legacy Migration)

Где: src/core/bridge.py — LegacyAllocatorBridge, BridgeAppAction

Что: Постепенная миграция из legacy (SLM/Sanali) в BCor. Bridge пытается получить зависимость из dishka, затем из legacy Allocator.

CONCEPT

Recursive Context Tracing

Где: src/core/messagebus.py:55

Что: Каждое событие носит triggering_message_id, позволяя восстановить полную цепочку вызовов. Реализовано через trace_stack в базовом Message. Патч bubus для отключения его 2-уровневого loop prevention.

CONCEPT

Scraper Engine (Playwright + BS4)

Где: src/adapters/web/scraper_engine.py

Что: Модуль веб-скрапинга: IBrowser (Playwright) → IExtractor (BS4) → ResourceDownloader. Поддержка пагинации, загрузки ресурсов, дедупликации.
